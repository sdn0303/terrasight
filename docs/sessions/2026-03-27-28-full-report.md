## 4. WASM Phase 0+1 — 正確性 (Correctness)

> **背景**: WASM 空間エンジンに 3 つのシステム的問題: (1) ready が単一 boolean で partial load を検知できない、(2) query_layers が double-JSON 返却、(3) 各レイヤーが個別 worker roundtrip。Phase 0+1 で (1) を完全解決。

| Task | 内容 | 変更ファイル | Commit |
|------|------|-------------|--------|
| **Task 0** | `canonicalLayerId()` — UI (underscore) と WASM/FGB (hyphen) の ID 正規化 | `layer-ids.ts` (new), `spatial-engine.ts` | `79351df` |
| **Task 1** | `loadedLayers: Set<string>` — 単一 boolean `_ready` を per-layer 追跡に置換。`queryReady()` API 追加 | `spatial-engine.ts`, `use-spatial-engine.ts` | `656ee4d` |
| **Task 2** | `query-error`/`stats-error` — リクエスト単位エラー分離。catch-all は init-level のみに限定 | `spatial-engine.ts`, `worker.ts` | `6adaf75` |
| **Task 3** | Performance API 計装 — `wasm-init`, `layer-load-{id}`, `wasm-query-{id}` の mark/measure | `spatial-engine.ts`, `worker.ts` | `291a6ba` |

**設計決定**: WASM stats は Phase 1 で無効化継続。6 レイヤー中 5 つが API-only データのため、FGB だけでは stats parity 不可。Backend `/api/stats` が canonical source。

## 5. WASM Phase 2 — 境界最適化 (Boundary Optimization)

> **背景**: Phase 0+1 完了後の最大ボトルネック: (1) `query_layers` の double JSON serialization、(2) N 個の static layer component がそれぞれ独立に worker roundtrip を発行。

### 5.1 Task 2.1 — Rust JSON Flattening

| 項目 | Before | After |
|------|--------|-------|
| `query_layers_inner` 返却型 | `HashMap<&str, String>` | `HashMap<&str, serde_json::Value>` |
| 使用メソッド | `get_features_geojson()` (手動文字列連結) | `get_features_as_value()` (構造化 Value) |
| Wire format | `{ "geology": "{\"type\":\"FeatureCollection\",...}" }` | `{ "geology": {"type":"FeatureCollection",...} }` |

**ファイル**: `services/wasm/src/lib.rs` (3 箇所: method, doc comment, test)

### 5.2 Task 2.2 — JS Single JSON.parse

| 項目 | Before | After |
|------|--------|-------|
| Parse 回数 | 2 (outer + inner per layer) | 1 (outer のみ) |
| Inner parse | `JSON.parse(raw)` for each string value | 不要 (values は object) |

**ファイル**: `services/frontend/src/lib/wasm/spatial-engine.ts`

### 5.3 Task 2.3 — `queryPerLayer()` メソッド

既存の `query()` (merged FC 返却) を維持しつつ、新規 `queryPerLayer()` メソッドを追加:

```typescript
async queryPerLayer(bbox: BBox, layers: string[]): Promise<Map<string, FeatureCollection>>
```

- 既存 worker message protocol を再利用（新 message type 不要）
- `PendingPerLayerQuery` 型を pending union に追加
- `query-result` handler で `pending.kind` に応じて merge / per-layer を分岐
- `isFeatureCollection()` 型ガード関数を抽出

**ファイル**: `services/frontend/src/lib/wasm/spatial-engine.ts`

### 5.4 Task 2.4 — `useVisibleStaticLayers` Hook

2-layer cache architecture の batched hook を新規作成:

| Cache Layer | 用途 | TTL | bbox 依存 |
|-------------|------|-----|----------|
| Layer 2 (WASM) | `queryPerLayer()` viewport intersection | 5s | Yes |
| Layer 1 (FGB fallback) | Full FlatGeobuf load (WASM 未 ready 時) | Infinity | No |

```typescript
export function useVisibleStaticLayers(
  visibleLayerIds: string[],
): Map<string, FeatureCollection>
```

- `useQuery` (WASM batch) + `useQueries` (FGB fallback) の組み合わせ
- `canonicalLayerId()` で ID 正規化
- `spatialEngine.queryReady()` で WASM/fallback を分類

**ファイル**: `services/frontend/src/hooks/use-visible-static-layers.ts` (新規)

### 5.5 Task 2.5 — Static Layer Component Migration

13 コンポーネントを self-fetch → prop-receive パターンに移行:

| Before | After |
|--------|-------|
| 各コンポーネントが `useStaticLayer()` を独立呼出 | `layer-renderer.tsx` が `useVisibleStaticLayers()` で一括取得し `data` prop で配布 |
| N worker roundtrips / viewport change | 1 worker roundtrip / viewport change |

**移行パターン**:
```tsx
// Before
export function GeologyLayer({ visible }: Props) {
  const { data } = useStaticLayer("13", "geology", visible);

// After
export function GeologyLayer({ visible, data: propData }: Props) {
  const selfFetch = useStaticLayer("13", "geology", visible && !propData);
  const data = propData ?? selfFetch.data;
```

**対象コンポーネント** (13 files):
`geology`, `landform`, `admin-boundary`, `did`, `fault`, `flood-history`, `liquefaction`, `railway`, `seismic`, `soil`, `volcano`, `station`, `landslide`

`useStaticLayer` に `@deprecated` JSDoc を追加。フォールバックとして機能継続。


## 10. 学んだこと (Learnings)

### サブエージェント失敗インシデント
- **事象**: Phase 2 Task 1 の Rust+JS cross-system タスクを sonnet 1 エージェントに委譲 → 70 分間 Read 44 回で停滞
- **根因**: Rust と JS の変更を 1 タスクに詰め込み、コンテキスト不足で繰り返し読み込み
- **対策**: `.claude/rules/workflow.md` に anti-pattern 追記。言語境界で必ず分割
- **本セッションでの再発**: `rust-engineer` agent が context compaction ループに陥り停滞。直接編集に切替えて 3 分で完了

### Codex Review 統合
- Design spec と implementation plan は別ラウンドで Codex レビューに出すと効果的
- レビュー指摘は修正済みを主張するだけでなく行番号レベルで evidence を示す

### 2-Layer Cache 設計
- FGB asset cache (infinite TTL) と viewport query cache (5s TTL) を混ぜると pan 時に不要な re-fetch が発生
- `useQuery` と `useQueries` を分離し、cache semantics を明確に分けることで解決

### Wire Format 変更の協調
- Rust 側の Value 化と JS 側の single parse は同時デプロイ必須
- 本セッションでは Task 2.1 → 2.2 を連続実行し、中間状態でのテスト失敗を回避
