## Vision

### 10x Check
現状の14レイヤーは「表示できるデータを並べた」状態。10x版は「投資判断に必要なデータを網羅し、交差分析で独自インサイトを生む」プラットフォーム。具体的には:
- 駅別乗降客数 × 地価トレンド × 将来推計人口 = 「賃貸需要予測マップ」
- 学区 × 公園 × 医療 × 犯罪 = 「居住快適性ヒートマップ」
- 全災害レイヤー統合（洪水+津波+土砂+液状化+断層+火山）= 「複合災害リスクインデックス」
- 立地適正化計画 × 都市計画道路 × 再開発計画 = 「行政投資シグナルマップ」

### Delight Opportunities
1. 駅マーカーのサイズが乗降客数に比例 → 「駅力」が一目でわかる
2. 学区境界をタップすると学区内の平均地価・人口推計が表示
3. 将来人口メッシュを赤(減少)→青(増加)のヒートマップで表示
4. 土砂災害レッドゾーンに赤い3D extrusion（「ここは危険」の視覚的インパクト）
5. 立地適正化計画区域に緑のオーバーレイ（「行政が推している」シグナル）

## Error Handling & Fallback Strategy

| Failure | Detection | Fallback | User Sees |
|---------|-----------|----------|-----------|
| GeoJSON fetch 404/timeout | fetch response status | Skip layer, log warning | Layer toggle disabled + "データ読込エラー" tooltip |
| Malformed GeoJSON (parse error) | try/catch on JSON.parse | Skip layer, report to console | Same as above |
| Oversized mesh data (>10MB) | Build-time file size check | process-geodata.pyでジオメトリ簡略化 (tolerance=0.0001) して10MB以下に収める。超過時はtippecanoeで.pmtiles変換 | N/A (build-time制御) |
| reinfolib API unavailable | fetch timeout (5s) | Use cached data or demo mode | "APIに接続できません。デモデータを表示中" |
| reinfolib API rate limit (429) | Response status | Exponential backoff (3 retries) | Brief loading indicator |
| process-geodata.py SHP read error | Exception in pyshp | Log error + skip dataset + continue others | N/A (build-time only) |

**Static layer fallback policy:** 静的GeoJSONのfetch失敗時はフォールバックデータなし。トグル無効化 + エラーtooltipで対応。理由: 静的ファイルは`public/geojson/`に同梱されるため、CDN障害以外でのfetch失敗は想定しにくい。

## Data Loading Strategy (Phase 1: 14→21, Phase 2: →22 layers)

**問題:** Phase 1で21、Phase 2で22レイヤーに拡大。全てを起動時にfetchするとメモリ・帯域を圧迫する。

**戦略: Lazy Load on Toggle**
1. 起動時はvisible=trueのレイヤーのみfetch（デフォルト: landprice + zoning）
2. ユーザーがトグルONにした時点で初めてfetch
3. 一度fetchしたデータはメモリキャッシュ（LRU方式: 合計40MB超過時に最も古い非表示レイヤーを解放）
4. zoom < minZoomのレイヤーはデータをfetchしてもsource上は非表示

## メッシュデータ仕様 (#5 将来推計人口)

**データソース:** 国立社会保障・人口問題研究所（NIPSSR）「日本の地域別将来推計人口」250mメッシュ
- ダウンロード: e-Stat 統計GIS または MLIT 国土数値情報 (M05005)
- フォーマット: Shapefile (SHP+DBF+SHX) — 3次メッシュコード体系
- 座標系: JGD2011 (EPSG:6668) → WGS84 (EPSG:4326) 変換必要

**process-geodata.pyでの処理パターン:**
1. SHPファイル読込 (pyshp)
2. Tokyo bbox `TOKYO_BBOX = [138.9, 35.4, 140.0, 35.95]` でフィルタ（既存process-geodata.pyと同一定数を使用）
3. メッシュコード → ポリゴン変換: `jismesh`ライブラリ使用（3次メッシュ = 緯度30秒×経度45秒 ≈ 約250m四方）。CRS変換は`pyproj` (EPSG:6668→4326)
4. 年次別人口カラム (2020実績, 2025, 2030, 2035, 2040, 2045, 2050推計) を保持
5. GeoJSON出力 (features配列、各featureに全年次のpropertiesを含む)

**既存パターンとの違い:** 従来のSHP→GeoJSONは単純なポリゴン/ポイント変換だが、メッシュデータはグリッド座標計算が必要。process-geodata.pyに`convert_mesh_population()`関数を追加。

**モバイル対応:** 50,000ポリゴンはモバイル端末でGPU負荷が高い。対策: モバイル(viewport < 768px)ではminZoomを14に引き上げ、表示ポリゴン数を制限。Performance Budgetの30 FPS基準を維持。

**ファイルサイズ制御:** メッシュの矩形ジオメトリは頂点が少なく、geometry simplificationは効果が薄い。サイズ超過時はproperties削減（中間年次を削除し、2020実績・2050推計・増減率のみ保持）を優先。それでも10MB超過の場合はtippecanoeで.pmtiles変換。process-geodata.pyに出力ファイルサイズアサーション (≤10MB) を組み込む。

## Rust WASM アーキテクチャ (フロントエンド計算エンジン)

### 動機
フロントエンド側でリアルタイムに必要となる以下の計算処理は、JavaScript実装ではパフォーマンスボトルネックとなる:
- **座標計算:** メッシュコード↔座標変換、bbox内ポイント判定、距離計算
- **境界判定:** Point-in-Polygon判定（「この物件はどの学区/浸水区域/立地適正化区域内か？」）
- **スコアリング:** 投資スコア算出（4コンポーネント: 地価トレンド、災害リスク、利便性、将来性）
- **データフィルタリング:** 30,000+地価ポイントの年度フィルタ、50,000+メッシュの増減率計算
- **交差分析:** 複数レイヤーの空間クロス集計（駅×地価×人口の相関）
