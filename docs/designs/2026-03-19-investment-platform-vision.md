## 1. エグゼクティブサマリー

### 現状
国交省のGeoJSONデータを3Dダークマップ上に表示する**ビューアー**プロトタイプ。
7レイヤー（地価公示、用途地域、液状化、洪水、急傾斜地、学校、医療機関）をモックデータで描画。

### ビジョン
「**不動産投資の意思決定プラットフォーム**」— 地図上の任意のポイントをクリックすると、0.5秒以内に投資スコア、防災リスク、周辺施設、地価トレンドを統合した「投資レポート」が表示される。投資家が「10秒で投資判断ができる」と感じるUX。

### ターゲット
- 個人不動産投資家（国内50万人+）
- 不動産仲介業者のデューデリジェンス支援
- 将来: 機関投資家向けAPI

### ビジネスモデル
- フリーミアム（基本閲覧無料 + プレミアム機能月額）
- B2B API（機関投資家・不動産テック企業向け）

### 地図ライブラリ: MapLibre GL JS（Phase 1）→ Mapbox GL JS（Phase 2以降で検討）

**Phase 1 は MapLibre 維持**（エンジニアリングレビューの推奨に基づく）。
- Globe view / terrain 3D は現時点で不要
- 従量課金リスクなし
- react-map-gl が両対応のため、切替は3箇所の変更のみ（後からでも低リスク）

**Phase 2以降で Mapbox 切替を検討する条件**:
- Globe view / terrain 3D が必要になった場合
- Mapbox Studio でカスタムスタイルを作り込む必要が出た場合
- 大量 Feature 描画の性能がMapLibreで不足した場合

### バックエンド言語選定理由（Python FastAPI → Rust Axum）

| 観点 | Python FastAPI | Rust Axum | 判断 |
|------|---------------|-----------|------|
| 全国GeoJSON bboxフィルタ（数GB） | GIL制約、メモリ高 | ゼロコピーパース、並列処理 | **Rust圧勝** |
| 投資スコア空間計算 | shapely経由、遅い | `geo` crateでネイティブ | **Rust圧勝** |
| PostGIS非同期クエリ | asyncpg — 十分速い | sqlx — 同等〜やや速い | 同等 |
| GeoJSON serialize | json.loads — GB級で秒単位 | serde_json — 桁違い | **Rust圧勝** |
| メモリ使用量 | 高い（GC + object overhead） | 1/5〜1/10 | **Rust圧勝** |
| 同時リクエスト | GIL制約 | tokioワークスティーリング | **Rust優位** |
| デプロイサイズ | Python + deps = 数百MB | シングルバイナリ ~10MB | **Rust優位** |
| 開発速度 | 速い | やや遅い | Python優位 |

→ このプロジェクトのホットパス（空間計算 + 大量GeoJSON処理）はRustが最適。

## 4. 投資スコアリングエンジン設計

### スコア構成

```
投資スコア (0-100) =
    地価トレンドスコア (0-25)     × 重み
  + 防災リスクスコア (0-25)       × 重み（反転: 低リスク = 高スコア）
  + 施設アクセススコア (0-25)     × 重み
  + 利回りポテンシャル (0-25)     × 重み
```

### 各要素の算出方法

| 要素 | データソース | 算出方法 |
|------|-----------|---------|
| 地価トレンド | L01 × 複数年度 | 5年間のCAGR（年平均成長率） |
| 防災リスク | 液状化 + 洪水 + 急傾斜地 | 半径500m内のリスクポリゴン重畳率 |
| 施設アクセス | 学校 + 医療 + 駅 | 半径1km内の施設数・距離加重 |
| 利回り | 取引価格 / 地価 | 周辺取引価格と公示地価の比率 |

### PostGISクエリ例（sqlxから呼び出し）

```rust
// 半径500m内の液状化リスクポリゴン重畳率
let overlap: f64 = sqlx::query_scalar(r#"
    SELECT COALESCE(
        SUM(ST_Area(ST_Intersection(
            ST_Buffer(ST_SetSRID(ST_MakePoint($1, $2), 4326)::geography, 500)::geometry,
            geom
        ))) / ST_Area(ST_Buffer(ST_SetSRID(ST_MakePoint($1, $2), 4326)::geography, 500)::geometry),
        0.0
    )
    FROM liquefaction_risk
    WHERE ST_DWithin(geom::geography, ST_SetSRID(ST_MakePoint($1, $2), 4326)::geography, 500)
"#)
.bind(lng)
.bind(lat)
.fetch_one(&pool)
.await?;
```

```
         + /api/stats（エリア統計）
         + FEのスコアカード・ダッシュボード対応

Week 4:  #4 比較分析パネル + #8 地価推移Sparkline
         + セキュリティ: bbox制限、CORS明示設定、レート制限

Week 5+: #6 全国対応（まず首都圏4県で性能検証 → 段階的拡大）
```

> **エンジニアリングレビューからの変更点**:
> - Python BEへの新機能追加を廃止（Rustに直接実装）
> - Rust移行を2週間に拡大（Week 1: scaffold, Week 2: データ投入 + API）
> - テストをRust移行と並行で追加（最小限: health + layers）
> - セキュリティ対策をWeek 4に前倒し
> - 全国対応は首都圏4県で性能検証してから段階的に拡大
> - **APIエンドポイントパスは `/api/area-data` を維持**（FE変更なし）

## 7. リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| reinfolib APIキー発行遅延 | Phase 2ブロック | 国土数値情報で先行開発 |
| PostGIS全国データの性能 | レイテンシ増加 | 空間インデックス + マテリアライズドビュー。**受入基準: bbox P99 < 100ms** |
| フロントエンド大量Feature描画 | FPS低下 | MapLibreクラスタリング + viewport culling |
| 投資スコアの精度 | ユーザー信頼性 | 「参考値」表記 + 計算根拠の透明性 |
| 認証基盤の選択 | Phase 2の工数 | 後日決定。ただし JWT vs セッションの方向性はPhase 1で決定 |
| Rust Axum 移行の開発コスト | **2週間**のバックエンド書き換え | 現FastAPIが~200行と小規模、APIインターフェース不変 |
| Rust クレートの成熟度 | geo/geozero は十分安定 | crates.io DL数・メンテナンス状況確認済み |
| XKT025/026（液状化/洪水）の存在不確実性 | reinfolib APIで取得できない可能性 | 国土数値情報A31(洪水) + 東京都建設局SHP(液状化)をPostGISに直接投入 |
